## v2：解決 bubble_sort.c 寫入失敗

### 問題

AI 使用 `printf` 或 `cat << EOF` 寫檔時：
1. `printf` 內的 `%d` 被 shell 當作 format specifier，導致錯誤
2. Windows cmd.exe 不支援 `<<` heredoc 語法
3. `extract_paths_from_command()` 誤抓 printf 引號內內容（如 `\n\nvoid`）為路徑

### 解決方案

| 修改 | 說明 |
|------|------|
| `extract_paths_from_command()` | 先移除引號內內容再匹配，避免 format string 誤判 |
| System prompt | 引導 AI 用 `write_file` 工具取代 shell 寫檔 |
| 新增 `write_file` 工具 | Python 原生 `open().write()`，繞過 shell 完全避免轉義問題 |

## v3：模組拆分

### 問題

單一 `agent0.py` 達 442 行，維護困難。

### 解決方案

拆分為 `src/` 套件，共 6 個模組：

```
05/
├── agent0.py              ← 入口點（3 行）
├── src/
│   ├── __init__.py
│   ├── config.py          ← 設定常數
│   ├── security.py        ← 安全審查
│   ├── gemini.py          ← Gemini API 封裝
│   ├── memory.py          ← 記憶管理
│   ├── tools.py           ← 工具定義與執行
│   └── agent.py           ← 主邏輯 + main()
├── _doc/
│   ├── 對話記錄.md
│   └── 程式架構與運行.md
└── README.md
```

### 關鍵技術決策

- 使用相對匯入（`from .config import ...`）避免套件路徑問題
- `PROGRAM_DIR` 向上取兩層（`src/config.py` → `src/` → `05/`）確保安全區域正確
- 無環狀依賴，各模組職責單一

## 已知問題

### Gemini API 429 配額錯誤

`RESOURCE_EXHAUSTED` — 免費 tier 配額用盡。

解法：
- 等待約 60 秒
- 改用其他 Gemini 模型（配額獨立計算）
- 改用 Ollama 本機 API（無配額限制）

### 環境變數讀取問題

在 CMD 有設但 VS Code PowerShell 讀不到。

解法：
- 完整重開 VS Code
- 或在 VS Code PowerShell 手動設 `$env:GEMINI_API_KEY="..."`

### Windows cmd.exe 限制

`subprocess.run(shell=True)` 在 Windows 使用 `cmd.exe`，不支援 `<<`、`'` 單引號等 bash 語法。

解法：寫檔一律使用 `write_file` 工具（Python 原生），不走 shell。
